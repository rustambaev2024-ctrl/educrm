# EduCRM — Claude Agent Instructions

## Проект
- Frontend: React + TypeScript + Vite (папка: корень проекта)
- Backend: Django + DRF (папка: backend/)
- БД: PostgreSQL (Railway)
- Деплой: Railway (автодеплой из ветки main)
- Репозиторий: https://github.com/rustambadev2024-ctrl/educrm

- admin — администратор

## 8 столпов UI — весь интерфейс строится по ним
Активные скилы: `frontend-design`, `design-system`, `shadcn`. Стек: TanStack Start + Tailwind v4 + shadcn/ui (new-york, radix, lucide). Всегда используй семантические токены (`bg-primary`, `text-muted-foreground`), `gap-*` вместо `space-y-*`, `size-*` вместо `w-* h-*`.

1. **Микроанимации и переходы.** Каждое действие имеет плавный отклик 150–300ms: карточки fade-in, панели выезжают, кнопка при нажатии «проваливается». Анимации shadcn Dialog/Sheet использовать как есть. Уважать `prefers-reduced-motion`.
2. **Типографика с иерархией.** С первого взгляда ясно: главное / второстепенное / подсказка. Разные размеры и веса (light/regular/semibold/bold), правильный line-height. Не делать всё одним размером.
3. **Состояния каждого элемента.** Минимум 5 состояний: обычное, hover, active, disabled, loading. Пользователь всегда понимает, можно ли кликнуть и что происходит.
4. **Пустые состояния.** Пустой список = компонент `Empty`: иконка + понятный текст («У вас пока нет студентов») + кнопка действия. Никогда просто белый экран.
5. **Обратная связь на каждое действие.** Клик → спиннер/«Сохраняем…» → `sonner` toast: зелёный «Сохранено» или красный с конкретной причиной. Ошибки прокидывают детали через `apiErrorMessage`, не голое «Xatolik».
6. **Детали для внимательного взгляда.** Аватар с инициалами — цвет детерминирован по имени, иконки выровнены с текстом (`data-icon`), чередование строк таблицы, тонкая цветная черта слева у активного пункта меню.
7. **Осмысленные цвета, не декоративные.** Красный = только опасность/удаление, зелёный = успех/активность, жёлтый = предупреждение, серый = неактивное. Цвет несёт информацию. Никаких `bg-blue-500` «потому что синий».
8. **Отзывчивость под контекст.** Не сжатый десктоп: на мобиле таблица → карточки, боковое меню → нижняя навигация, тач-таргеты ≥44px, модалки открываются снизу как `Drawer`/`Sheet`. student/ и parent/ порталы — mobile-first (`MobileLayout`).

### Подтверждения
Опасные и необратимые действия (удаление, отмена, возврат) подтверждаются через `ConfirmDialog` (`src/components/ui/confirm-dialog.tsx`) — НИКОГДА `window.confirm`.

## ТЕКУЩИЙ СТАТУС И ОТКРЫТЫЕ ЗАДАЧИ
(обновлено 2026-07-10, по итогам диагностики продакшна)

### Ветка
Рабочая копия сейчас на `staging` (up to date with `origin/staging`), не `master`.
Перед любым hotfix-коммитом уточнять, какая ветка реально деплоится в Railway
(сервис `educrm`, проект `pleasing-trust`) — не полагаться на инструкции по памяти.

### Жалобы пользователей (в разборе)
1. **Посещаемость не сохраняется у части учителей уже неделю.**
   Проверено 2026-07-10: гарда "группа должна быть active" в
   `backend/apps/lessons/views.py` **не существует** — ни в бэкенде, ни во
   фронтенде, ни в тестах (grep по всему репо не нашёл). Единственные реальные
   проверки в `attendance()`: `lesson.status == "cancelled"` и лимит
   бэкдейтинга 14 дней для teacher/support_teacher. Значит причина жалобы не
   в этом. Вероятные кандидаты, требуют проверки в БД:
   - `hasattr(user, "staff_profile")` — если у teacher-пользователя нет
     `staff_profile`, `get_queryset` молча возвращает `qs.none()` без ошибки
     (views.py, LessonViewSet/GroupViewSet).
   - Рассинхрон `branch_id`: `GroupViewSet.get_queryset` для teacher фильтрует
     и по `branch_id` учителя, и по `teacher=staff_profile` — при переводе
     учителя между филиалами группа перестаёт быть видна.
   - Лимит бэкдейтинга 14 дней мог неожиданно задеть старые уроки.
   - **Не чинить, добавляя гард по group.status "для надёжности" — его никогда
     не было, это создаст новый баг, а не решит текущий.**
2. **Учитель не видит свою группу в посещаемости** — та же причина, см. выше
   (staff_profile / branch_id).
3. **Группы не показываются** — требует отдельного разбора, не исследовано.

### Найденные при диагностике риски (не исправлены, только задокументированы)
- **Дублирование Celery worker/beat**: `docker/start.sh` сам поднимает
  `celery worker` + `celery beat` в фоне ВНУТРИ контейнера `educrm`
  (start.sh:35-41), при этом в репо есть отдельные `railway.worker.toml` /
  `railway.beat.toml` с теми же командами. Через Railway MCP подтверждено,
  что отдельных сервисов worker/beat в проекте `pleasing-trust` сейчас нет —
  но конфиги в репо вводят в заблуждение и рискованны при следующем деплое
  (риск двойных periodic-задач: `daily_lesson_charge`, SMS-рассылки).
- **`Group.status` (`recruiting/active/frozen/completed`) нигде не влияет на
  бизнес-логику посещаемости/уроков** — возможно, стоит явно решить, нужен ли
  такой гард вообще, вместо того чтобы добавлять его реактивно.
- Единый Redis (db/0) на cache + Celery broker/backend + Channels layer —
  потенциальная точка конкуренции под нагрузкой.
- `/media/logos/logo_*.jpg` периодически 404 — Railway ephemeral filesystem,
  вероятно нужен persistent volume или S3 для media.
- `/api/v1/attendance/` отдаёт большой ответ (~39KB в одном запросе в логах) —
  стоит проверить пагинацию.
